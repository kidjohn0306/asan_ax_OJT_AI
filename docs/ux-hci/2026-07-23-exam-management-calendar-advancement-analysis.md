# 시험 관리·달력 고도화 분석

## 1. 결론

달력을 세 화면에 각각 추가하는 것만으로는 충분하지 않다.
현재 더 큰 문제는 시험지 원본, 실제 시험 회차, 일정 상태가 하나의 `exam_sets` 흐름에 섞여 있고 각 화면이 `exam_datetime`을 서로 다르게 해석한다는 점이다.

권장 방향은 다음과 같다.

1. 시험지는 문항 구성과 대상 팀을 확정하는 원본으로 유지한다.
2. 시험 회차는 선택한 시험지 버전, 일정, 시간, 합격 기준, 응시자 배정을 갖는 운영 단위로 분명히 다룬다.
3. 팀은 시험지에서 상속하며 시험 회차 생성 화면에서는 임의로 바꾸지 않는다.
4. 시험지 선택은 상세 팝업으로 바꾸되 팀 필터와 버전 정보를 우선 제공한다.
5. 대시보드, 시험 생성·관리, 응시 현황은 같은 일정 데이터를 사용하지만 달력의 역할은 서로 다르게 둔다.
6. 일정 상태와 충돌은 장기적으로 서버가 판정하며 프런트엔드의 현재 시각 기반 추정은 제거한다.

## 2. 현재 도메인 구조

| 개념 | 현재 식별자와 필드 | 실제 역할 | 문제점 |
|---|---|---|---|
| 시험지 원본 | `exam_set_id`, `exam_id`, `team_code`, `question_ids` | 팀별 문항과 배점을 확정한다. | 시험 회차와 같은 저장소와 응답 목록에 섞인다. |
| 시험지 버전 | `paper_version`, `exam_version_id` | 동결된 문항·배점 버전을 가리킨다. | 시험 생성 UI는 특정 버전보다 `exam_set_id`만 선택한다. |
| 시험 회차 | 새 `exam_id`, 기존 `exam_set_id`, `exam_datetime`, `duration_min` | 원본 시험지를 특정 일정으로 실행한다. | 원본과 회차를 명시적으로 구분하는 필드가 약하다. |
| 대상 팀 | `team_code` | 출제 범위와 배정 가능 사용자를 제한한다. | UI에서 팀 고정 이유가 충분히 설명되지 않는다. |
| 응시자 배정 | `assigned_users` 또는 정규화 `assignments` | 시험 회차에 응시자를 연결한다. | 일정 충돌과 가용 기간 검사가 화면에 드러나지 않는다. |
| 시험 상태 | 클라이언트가 일시와 시간으로 계산 | 예정, 진행 중, 완료를 표시한다. | 화면마다 미정 일시 처리와 상태 이름이 다르다. |

현재 구현의 중요한 사실은 `exam_set_id`가 시험지 계보이고 `exam_id`가 개별 저장 행 또는 시험 회차라는 점이다.
시험 회차 생성 API는 원본의 `team_code`, 문항, 배점, 시험 유형을 그대로 상속한다.
응시자 배정 API도 응시자 팀과 시험 팀이 다르면 차단한다.

따라서 현재 구조에서 시험 회차의 팀을 별도 선택하게 만드는 것은 적절하지 않다.
화면만 자유롭게 바꾸면 서버 배정 규칙과 충돌하고, 팀별 출제 문항의 의미도 깨진다.

## 3. 확인된 구조적 문제

### 3.1 시험지 원본과 시험 회차가 목록에서 섞일 수 있다

시험지 저장과 시험 회차 저장이 모두 `exam_sets`를 사용한다.
원본 시험지는 일정이 없는 채 저장되고, 시험 회차는 같은 `exam_set_id`를 공유하며 새 `exam_id`를 갖는다.

`GET /api/admin/exam-sets`는 두 종류를 별도 분류하지 않고 모두 반환한다.
그 결과 일정이 없는 시험지 원본이 시험 관리나 대시보드에서 예정 시험처럼 보일 수 있다.

1차 개선에서도 최소한 화면 응답에 다음 구분이 필요하다.

- `record_type=paper`
- `record_type=exam_round`

장기적으로는 시험지 목록과 시험 회차 목록을 별도 조회 계약으로 나누는 편이 안전하다.

### 3.2 시험지 대표 버전 선택이 명확하지 않다

시험지 목록 API는 같은 `exam_set_id`를 가진 행을 묶어 대표 한 건을 반환한다.
그러나 시험 회차 생성 API는 전달받은 `exam_set_id`와 일치하는 첫 행을 원본으로 선택한다.

팝업에서 최신 시험지 버전을 봤더라도 실제 생성이 같은 대표 버전을 사용한다는 계약이 분명하지 않다.
시험지 선택 결과는 다음 식별자를 함께 가져야 한다.

- 시험지 계보 식별자 `exam_set_id`.
- 선택한 불변 버전 식별자 `exam_version_id`.
- 화면 상세 조회용 대표 행 식별자 `paper_exam_id`.

회차 생성 API도 선택한 버전을 명시적으로 전달받거나, 서버가 현재 버전을 단일 규칙으로 결정하고 응답에 반환해야 한다.

### 3.3 사용 횟수 의미가 모호하다

현재 `used_by_exam_count`는 같은 `exam_set_id`를 가진 행 수다.
원본 시험지 자신도 포함될 수 있어 실제 시험 회차가 하나도 없어도 사용 1회처럼 해석될 여지가 있다.

팝업과 목록에서는 다음 값을 구분하는 편이 낫다.

- 생성된 시험 회차 수.
- 예정 회차 수.
- 완료 회차 수.
- 마지막 사용 일시.

### 3.4 상태 판정 규칙이 화면마다 다르다

대시보드와 시험 관리의 `getExamStatus`는 일시가 없거나 잘못된 경우 `예정`으로 반환한다.
응시 현황의 `getScheduledExamStatus`는 같은 경우 `일정 미정`으로 반환한다.

또한 모든 상태는 브라우저 시간과 `exam_datetime + duration_min`으로 계산된다.
관리자 PC 시간, 시간대, 문자열 형식에 따라 결과가 달라질 수 있다.

서버 응답에 다음 값을 제공하는 방향을 권장한다.

- `scheduled_start_at`.
- `scheduled_end_at`.
- `timezone`.
- `schedule_status`.
- `lifecycle_status`.
- `can_reschedule`.
- `schedule_warning_codes`.

기존 V2 스키마에는 이미 `scheduled_start_at`, `scheduled_end_at`, `lifecycle_status`, `live_status`가 있어 이 방향과 맞는다.

### 3.5 일정 입력과 수정에 서버 검증이 부족하다

현재 일정 API는 문자열을 받고 그대로 저장한다.
다음 검증이 없다.

- 유효한 날짜 형식인지 확인.
- `Asia/Seoul` 기준인지 UTC인지 확인.
- 종료 시각이 시작 시각보다 뒤인지 확인.
- 같은 팀의 다른 시험과 겹치는지 확인.
- 이미 시작하거나 완료된 시험을 수정하는지 확인.
- 응시자가 배정된 시험의 변경인지 확인.
- 동시 수정으로 최신 값이 덮어써지는지 확인.

달력 UI보다 먼저 또는 같은 단계에서 이 규칙의 최소 계약을 확정해야 한다.

### 3.6 응시 현황의 조회 방식이 시험 수에 비례해 커진다

응시 현황은 10초마다 시험 목록을 한 번 조회한 뒤 각 시험의 결과를 별도로 요청한다.
시험이 N개면 한 주기마다 `1 + N` 요청이 발생한다.

달력으로 기간 조회까지 추가하면 이 구조는 더 무거워진다.
응시 현황 전용 집계 API가 필요하다.

예시는 다음과 같다.

`GET /api/admin/exams/live-summary?from=2026-07-01&to=2026-07-31&team=T1&status=in_progress`

이 응답은 시험별 배정, 제출, 미제출, 오류 수와 서버 계산 상태를 한 번에 반환해야 한다.

## 4. 화면별 달력 역할

세 화면에 같은 모양의 큰 달력을 반복하는 것은 권장하지 않는다.
같은 데이터 소스를 쓰되 사용자 목적에 맞춰 역할을 나눠야 한다.

### 4.1 대시보드 달력

역할은 요약과 빠른 진입이다.

현재 장점은 월 단위 일정과 날짜별 점 표시가 이미 있다는 점이다.
다만 클릭 가능한 `div`로 구현되어 키보드 접근성이 부족하고, 일정이 없는 날짜는 선택할 수 없으며, 일정 상세로 이동하는 기능도 없다.

권장 개선은 다음과 같다.

- 오늘과 향후 7일의 시험을 우선 강조한다.
- 일정 충돌, 미배정, 일정 미정 같은 운영 경고를 함께 보여준다.
- 날짜 클릭 시 시험 관리 화면의 해당 날짜로 이동한다.
- 시험명 클릭 시 `/admin/exams/:examId`로 이동한다.
- `시험 만들기` 진입은 `/admin/exams/create?date=YYYY-MM-DD`처럼 날짜를 전달한다.
- 대시보드에서는 일정 수정과 상세 필터를 제공하지 않는다.

대시보드는 완전한 일정 관리 도구가 아니라 운영 우선순위를 알려주는 관제 진입점이어야 한다.

### 4.2 시험 생성·관리 달력

역할은 일정 생성, 조회, 수정이다.

현재 상단의 한 줄 생성 폼은 시험지, 대상, 시험명, 일시, 시간, 커트라인을 모두 가로로 압축한다.
여기에 큰 달력을 그대로 추가하면 정보 밀도가 더 높아지고 생성 흐름이 불분명해진다.

권장 구조는 다음과 같다.

1. 상단에 `달력 보기`, `목록 보기` 전환을 둔다.
2. 기본 화면은 월 달력과 선택 날짜의 일정 패널로 구성한다.
3. `시험 만들기` 버튼 또는 빈 날짜 선택으로 생성 패널을 연다.
4. 생성은 시험지 선택, 일정 설정, 응시자 확인의 세 단계로 나눈다.
5. 일정 미정 시험은 달력 밖의 `미정 일정` 보관함에 둔다.
6. 목록 보기에서는 팀, 상태, 기간, 시험지로 필터링한다.
7. 필터와 선택 날짜는 URL query에 유지한다.

권장 URL 상태 예시는 다음과 같다.

`/admin/exams?view=calendar&date=2026-07-23&team=T1&status=scheduled`

### 4.3 응시 현황 달력

역할은 운영 대상을 날짜로 좁히는 필터다.

응시 현황의 핵심은 지금 응시 중인 시험과 오류를 빠르게 찾는 것이다.
큰 월 달력이 목록보다 더 많은 공간을 차지하면 운영 집중도가 떨어진다.

권장 구조는 다음과 같다.

- 기본값은 `오늘`이다.
- `오늘`, `이번 주`, `날짜 선택` 빠른 필터를 제공한다.
- 데스크톱에서는 축소 달력과 해당 날짜의 시험 목록을 함께 보여준다.
- 작은 화면에서는 달력을 팝오버 또는 드로어로 연다.
- 날짜 셀에는 예정, 진행 중, 완료 건수를 작은 배지로 표시한다.
- 오류는 상태 필터와 별도 경고 배지로 유지한다.
- 선택 날짜, 팀, 상태는 URL query에 저장한다.
- 시험 클릭 시 기존 시험별 상세 화면으로 이동한다.

응시 현황 달력은 일정을 수정하지 않는다.
일정 변경은 시험 생성·관리 화면으로 이동시켜 권한과 감사 범위를 분리한다.

## 5. 시험지 선택 팝업 권장안

### 5.1 팝업이 필요한 이유

현재 선택 상자는 이름, 팀, 문항 수만 한 줄로 보여준다.
시험지가 많아지면 이름이 비슷하거나 버전이 여러 개인 경우 잘못 선택하기 쉽다.

팝업은 단순히 더 큰 선택 상자가 아니라 선택 근거를 제공해야 한다.

### 5.2 권장 정보 구조

팝업 상단 필터는 다음과 같다.

- 팀.
- 시험 유형.
- 시험지명 또는 ID 검색.
- 사용 여부.
- 최신 버전만 보기.

중앙 목록은 다음 정보를 보여준다.

- 시험지명.
- 대상 팀.
- 시험 유형.
- 버전.
- 문항 수.
- 총점.
- 생성일.
- 생성자.
- 실제 시험 회차 수.
- 마지막 사용일.

오른쪽 상세 패널은 다음 정보를 보여준다.

- 문항 구성 요약.
- 난이도 분포.
- 카테고리 분포.
- 문항별 배점 합계.
- 최근 사용 회차.
- 선택한 불변 버전.
- `문항 상세 보기` 링크.

선택 버튼은 `이 시험지로 시험 만들기`처럼 결과를 명확히 표현한다.

### 5.3 팀 선택 로직

권장 원칙은 `시험지 팀 고정, 시험 회차 팀 상속`이다.

이유는 다음과 같다.

- 시험지의 문항이 팀별 교육자료와 출제 범위를 반영한다.
- 서버가 이미 다른 팀 응시자 배정을 차단한다.
- 회차에서 팀을 바꾸면 시험지 내용과 대상이 불일치할 수 있다.
- 팀 변경을 허용하면 감사와 결과 비교의 기준도 흔들린다.

생성 흐름은 다음과 같이 구성한다.

1. 관리자가 대상 팀을 먼저 고르거나 전체 시험지에서 팀 필터를 사용한다.
2. 팝업은 해당 팀의 시험지를 우선 보여준다.
3. 시험지를 선택하면 대상 팀은 읽기 전용으로 확정된다.
4. 응시자 선택 목록은 같은 팀의 승인 사용자만 활성화한다.
5. 다른 팀을 대상으로 하려면 다른 시험지를 선택하거나 수정본을 만든다.

### 5.4 공통 시험지가 필요할 때

여러 팀이 완전히 같은 시험지를 사용해야 하는 실제 요구가 있다면 팀 덮어쓰기로 해결하지 않는다.
별도 범위 모델을 도입하는 편이 안전하다.

예시는 다음과 같다.

- `scope_type=team`.
- `scope_type=common`.
- `eligible_team_codes`.

공통 시험지는 공통 문항만 포함하고 여러 팀 회차 생성이 가능해야 한다.
팀별 문항이 하나라도 포함되면 해당 팀으로 범위를 제한한다.

이 변경은 데이터 모델과 출제 규칙을 바꾸므로 달력 1차 고도화와 분리해야 한다.

## 6. 권장 시험 생성 흐름

### 1단계 시험지 선택

- 팀 필터를 선택한다.
- 시험지 팝업에서 버전과 구성을 확인한다.
- 선택한 시험지와 대상 팀을 요약 카드로 고정한다.

### 2단계 일정 설정

- 시작 일시.
- 시험 시간 또는 종료 일시.
- 시간대 `Asia/Seoul`.
- 합격 커트라인.
- 정식 또는 연습 구분.
- 일정 충돌 검사 결과.

달력에서 빈 날짜를 선택해 진입하면 날짜를 미리 채우고 시간만 입력하게 한다.

### 3단계 응시자 확인

- 같은 팀의 승인 사용자만 표시한다.
- 배정 인원과 제외 인원을 요약한다.
- 정식 시험 재배정 시 기존 정식 시험이 해제된다는 정책을 표시한다.
- 충돌 사용자와 미승인 사용자를 별도로 알려준다.

### 생성 전 최종 요약

- 시험지명과 버전.
- 대상 팀.
- 시험명.
- 시작과 종료 일시.
- 시험 시간.
- 합격 커트라인.
- 배정 인원.
- 충돌과 경고.

## 7. 일정 수명주기 권장안

| 상태 | 의미 | 허용 작업 |
|---|---|---|
| `draft` | 시험지는 선택했지만 일정이 확정되지 않았다. | 일정, 이름, 기준 수정과 삭제. |
| `scheduled` | 미래 일정이 확정됐다. | 규칙에 따른 재일정과 응시자 배정. |
| `in_progress` | 시험이 시작됐고 종료 전이다. | 조회만 허용하고 운영 제어는 후속 단계로 둔다. |
| `completed` | 정상 종료됐다. | 결과와 이력 조회만 허용한다. |
| `cancelled` | 일정이 취소됐다. | 이력 조회와 필요 시 새 회차 복제. |

현재 `status=active`와 브라우저 계산 상태만으로는 이 수명주기를 표현하기 어렵다.
1차에서는 서버 계산 `schedule_status`를 추가하고, 이후 `lifecycle_status`로 전환하는 방식이 안전하다.

## 8. 일정 충돌 정책 제안

권장 기본값은 다음과 같다.

- 같은 팀의 정식 시험 시간이 겹치면 생성을 차단한다.
- 같은 팀의 연습 시험이 겹치면 경고하되 정책에 따라 허용할 수 있다.
- 다른 팀의 시험이 겹치는 것은 허용한다.
- 같은 응시자가 여러 회차에 동시에 배정되면 차단한다.
- 시작한 시험은 일반 일정 수정으로 변경하지 않는다.
- 배정자가 있는 예정 시험의 일정 변경은 확인과 변경 사유를 요구한다.
- 완료 시험의 일정과 시험지는 수정하지 않는다.

충돌 응답에는 단순 오류 문자열이 아니라 겹치는 시험 ID, 이름, 시간, 팀, 충돌 유형을 포함해야 한다.

## 9. 접근성과 HCI 개선

현재 대시보드 달력 날짜는 클릭 가능한 `div`다.
키보드 사용자는 날짜를 선택할 수 없고 선택 상태도 보조기기에 충분히 전달되지 않는다.

공통 달력 컴포넌트는 다음을 지원해야 한다.

- 날짜를 실제 `button`으로 렌더링한다.
- 화살표 키로 날짜를 이동한다.
- `오늘`, `선택됨`, 시험 건수, 공휴일 이름을 접근 가능한 이름으로 제공한다.
- 선택 날짜와 현재 포커스를 시각적으로 구분한다.
- 색상 외에 상태 텍스트나 모양을 함께 사용한다.
- 일정이 3건을 넘으면 `외 N건`을 표시한다.
- 빈 날짜도 선택해 시험 생성을 시작할 수 있다.

## 10. 단계별 구현 권장 순서

### 0단계 제품 결정

- 팀 고정 원칙을 승인한다.
- 공통 시험지 필요 여부를 확인한다.
- 일정 충돌의 차단과 경고 정책을 확정한다.
- 월 보기만 먼저 할지 주 보기까지 포함할지 정한다.

### 1단계 프런트엔드 UX 계약

- 공통 달력 표시 모델을 정의한다.
- 시험 관리의 달력·목록 전환과 URL query를 정의한다.
- 시험지 선택 팝업의 필드와 선택 결과를 정의한다.
- 응시 현황 날짜 필터를 정의한다.
- 독립 목업으로 주요 흐름을 검증한다.

### 2단계 최소 API 보강

- 시험지와 시험 회차를 응답에서 구분한다.
- 시험지 버전 식별자를 회차 생성에 전달한다.
- 기간 기반 시험 일정 조회 API를 추가한다.
- 서버 계산 상태와 시간대를 반환한다.
- 일정 문자열 유효성 검사를 추가한다.

### 3단계 일정 무결성

- 시작과 종료 시각을 정규화한다.
- 일정 충돌 검사와 구조화된 오류를 추가한다.
- 재일정 제한과 변경 이력을 추가한다.
- 낙관적 잠금용 `row_version`을 적용한다.

### 4단계 응시 현황 집계

- `1 + N` 폴링을 단일 집계 API로 교체한다.
- 날짜, 팀, 상태 필터를 서버 조회에 연결한다.
- 마지막 정상 Snapshot 유지와 오류 복구 계약을 보존한다.

### 5단계 운영 확장

- 시작, 연장, 마감, 종료 같은 제어를 별도 승인 흐름으로 추가한다.
- 모든 제어에 멱등키, 사유, 감사 로그를 적용한다.
- 이 단계 전에는 응시 현황을 조회 전용으로 유지한다.

## 11. 1차 권장 범위

가장 현실적인 1차 범위는 다음과 같다.

- 시험 관리에 월 달력과 목록 전환을 추가한다.
- 빈 날짜에서 시험 생성 패널을 연다.
- 시험지 선택을 팀 필터가 있는 팝업으로 교체한다.
- 선택한 시험지 팀은 읽기 전용으로 유지한다.
- 대시보드 날짜와 시험을 시험 관리 상세로 연결한다.
- 응시 현황에 오늘, 이번 주, 날짜 선택 필터를 추가한다.
- 모든 필터와 선택 날짜를 URL query에 저장한다.
- 서버가 일정 형식과 시간대를 검증하고 상태를 반환한다.

1차에서 제외할 항목은 다음과 같다.

- 드래그 앤 드롭 일정 변경.
- 반복 시험.
- 다중 시간대.
- 강제 종료와 시간 연장.
- 공통 시험지의 다중 팀 범위.
- 외부 캘린더 연동.

## 12. 인수 기준

### 시험지 선택

- 시험지명, 팀, 유형, 버전, 문항 수로 구분해 선택할 수 있다.
- 선택 결과가 특정 시험지 버전을 가리킨다.
- 대상 팀은 선택한 시험지와 일치하며 임의 변경할 수 없다.
- 다른 팀 사용자는 배정 후보에서 비활성화되거나 제외된다.

### 시험 일정

- 빈 날짜에서 생성 흐름을 시작할 수 있다.
- 일정 미정 시험은 예정 시험과 구분된다.
- 같은 일정이 대시보드, 시험 관리, 응시 현황에서 같은 상태로 보인다.
- 시간대가 명시되고 서버가 잘못된 일정 문자열을 거부한다.
- 겹치는 일정은 정책에 따라 구조화된 경고 또는 차단으로 처리된다.

### 달력

- 월 이동, 오늘 이동, 날짜 선택을 키보드로 수행할 수 있다.
- 일정이 많은 날짜도 건수와 상세 목록을 확인할 수 있다.
- 팀과 상태 필터, 선택 날짜가 새로고침 후 유지된다.
- 대시보드 캘린더는 시험 관리로, 응시 현황 달력은 시험별 현황으로 연결된다.

### 운영 안정성

- 응시 현황 폴링 요청 수가 시험 개수에 비례해 증가하지 않는다.
- 최초 조회 오류와 후속 갱신 오류를 구분한다.
- 완료 시험은 일정과 시험지를 수정할 수 없다.
- 일정 변경은 변경자, 변경 시각, 변경 사유를 추적할 수 있다.

## 13. 구현 전 확인할 제품 질문

1. 공통 문항으로 만든 시험지를 여러 팀에서 그대로 사용하는 실제 사례가 있는가.
2. 한 팀에 같은 시간대의 정식 시험을 두 개 만들 가능성이 있는가.
3. 시험 일정 확정 전에 응시자를 먼저 배정할 수 있어야 하는가.
4. 배정 후 일정 변경 시 관리자 확인만 필요한가, 응시자 알림까지 필요한가.
5. 응시 현황의 기본 범위는 오늘인가, 진행 중 전체인가.
6. 1차 달력에서 월 보기만 제공해도 충분한가.

이 질문 중 1번과 2번이 확정되면 팀 및 충돌 모델을 고정할 수 있다.
나머지는 화면 단계와 운영 정책의 세부 범위를 결정한다.
