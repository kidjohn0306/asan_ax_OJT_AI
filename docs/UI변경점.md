# UI 변경점 명세 — 백엔드 MVP 설계 적용

| 항목 | 내용 |
|---|---|
| 참조 | [백엔드_구조_MVP설계.md](백엔드_구조_MVP설계.md) 1차 MVP 필수 9개 |
| 대상 파일 | `frontend/src/pages/Admin.jsx` · `frontend/src/pages/Exam.jsx` |
| 목업 | [UI목업_MVP.html](UI목업_MVP.html) |
| 원칙 | **기존 레이아웃·디자인 언어 유지.** 새 기능만 얹고 UX 일관성 깨지 않는다 |

---

## 0. 변경 요약 한 장

```
현재 UI                          →   MVP 적용 후 UI
────────────────────────────────────────────────────────────────
문제 관리: 난이도 변경만 가능        →  상태(status) 뱃지 + 승인·반려 액션
시험 생성: 전체 문제 미리보기        →  승인 문제만 출제 (approved 필터)
검토·수정: 하드코딩된 Mock 뷰        →  실제 검토 큐 (reviewing 상태 문제)
응시 이력: 하드코딩된 더미 데이터    →  results.jsonl API 연동
시험 결과: "데이터 삭제" 경고 표시   →  영속화 완료로 경고 제거
대시보드: 통계 카드 4개 (고정값)     →  승인 문제 수 · 검토 대기 수 추가
```

## 1. 관리자 화면 — 변경 상세

### 1-1. 문제 관리 뷰 (`Questions` 컴포넌트) — 핵심 변경

**현재**: 문항 목록 + 난이도 드롭다운만. `status` 개념 없음.

**변경 후**:

| 위치 | 변경 내용 |
|---|---|
| 필터 영역 | **상태 필터 탭** 추가: 전체 / 검토대기(`reviewing`) / 승인(`approved`) / 반려(`rejected`) |
| 문항 카드 | **상태 뱃지** 추가: `draft`(회색) / `reviewing`(노랑) / `approved`(초록) / `rejected`(빨강) |
| 문항 카드 | **플래그 아이콘** 추가: `warning`(⚠️) / `security_hold`(🔒) |
| 문항 카드 액션 | **승인 버튼** 추가 (reviewing 상태일 때만 활성) |
| 문항 카드 액션 | **반려 버튼** 추가 → 클릭 시 반려 사유 입력 인라인 폼 펼쳐짐 |
| 난이도 수정 | **사유 코드 선택** 추가 (드롭다운: 절차순서/기억단순/복합판단 등) |
| 해설 | **해설(`explanation`) 표시** 추가 (접기/펼치기) |

**API 변경**:
- 기존: `GET /api/admin/questions?category=`
- 추가: `?status=reviewing` 파라미터
- 추가: `POST /api/admin/questions/{id}/approve`
- 추가: `POST /api/admin/questions/{id}/reject` (body: `{ reason }`)
- 수정: `PATCH /api/admin/difficulty` → body에 `reason_code` 추가

---

### 1-2. 검토·수정 뷰 (`ExamReview` 컴포넌트) — 전면 교체

**현재**: 하드코딩된 Mock 뷰. 실제 API 연결 없음. 탭만 있고 문항은 1개 고정 표시.

**변경 후** (이 뷰가 실제 검토 큐로 작동):

| 위치 | 변경 내용 |
|---|---|
| 상단 탭 | `reviewing` 상태 문항만 표시. 탭을 `검토 대기 / 플래그 있음 / 오늘 처리됨` 으로 변경 |
| 문항 카드 헤더 | 게이트 검증 결과 뱃지 표시 (`PASS` / `WARNING` / `SECURITY_HOLD`) |
| 문항 카드 | 출처(`source.file_name`, `page`) 표시 (내부 자료 근거 확인용) |
| 문항 카드 | 해설(`explanation`) 항상 표시 (정답-해설 일치 사람이 직접 확인) |
| 난이도 수정 | **사유 코드 드롭다운 필수 선택** 후 변경 가능 |
| 액션 버튼 | `수정 요청` → `반려(사유 입력)` / `임시저장` 유지 / `승인` 유지 |
| 반려 사유 | 텍스트 입력 → **필수(빈값 제출 불가)** |
| 플래그 표시 | `warning` · `security_hold` 플래그 있으면 상단에 컬러 배너 표시 |

---

### 1-3. 시험 생성 뷰 (`ExamCreate` 컴포넌트) — 부분 변경

**현재**: 팀·난이도·문항수 선택 후 `preview-exam` API 호출 → 전체 문제 미리보기.

**변경 후**:

| 위치 | 변경 내용 |
|---|---|
| 생성 전 안내 | "승인된 문제 X개로 출제됩니다" 문구 + 승인 문제 수 API 조회 표시 |
| 미리보기 카드 | 각 문항에 **`approved` 뱃지** 표시 (승인 문제만 나옴을 시각적으로 확인) |
| 미리보기 카드 | `draft` / `reviewing` 문제가 섞이면 **경고 배너** 표시 (정상이면 안 뜸) |
| 하단 버튼 | `시험지 저장` 클릭 시 → 스냅샷 생성 완료 토스트 표시 |

---

### 1-4. 대시보드 (`Dashboard` 컴포넌트) — 통계 카드 변경

**현재**: 승인된 응시자 / 총 응시 완료 / 합격률 / 평균 점수 — 4개 카드.

**변경 후**: 카드 2개 교체 + 빠른 실행에 검토 큐 추가.

| 카드 | 현재 | 변경 |
|---|---|---|
| 1번 | 승인된 응시자 수 | **유지** |
| 2번 | 총 응시 완료 | **유지** |
| 3번 | 합격률 (60%) 하드코딩 | → **승인된 문제 수** (API: `approved` 필터 count) |
| 4번 | 평균 점수 (80.8) 하드코딩 | → **검토 대기 문항 수** (`reviewing` 필터 count) |

빠른 실행 버튼에 **`검토 큐`** 항목 추가 (`exam-review` 뷰로 이동).

---

### 1-5. 응시 이력 뷰 (`History` 컴포넌트) — API 연동

**현재**: `fetch_logs`가 하드코딩된 5개 더미 데이터 반환.

**변경 후**:
- 백엔드 `fetch_logs`가 `results.jsonl`에서 실제 데이터 읽어 반환
- UI 변경 없음 — 데이터만 실제화

---

### 1-6. 설정 뷰 (`Settings` 컴포넌트) — TODO 업데이트

**현재 TODO 목록 7개**에서 완료·변경된 항목 반영:

| 항목 | 변경 |
|---|---|
| `Google Drive Service Account 연동` | ✅ 완료 표시로 변경 |
| `Drive 결과로그 저장` | → `결과 JSONL 영속화 (in progress)` |
| `문제 상태 머신 (draft→approved)` | 신규 추가 |
| `규칙 게이트 7개 (V-01~V-07)` | 신규 추가 |
| `시험지 스냅샷 저장` | 신규 추가 |

## 2. 응시자 화면 — 변경 상세

### 2-1. 시험 결과 화면 (`ResultScreen`) — 경고 문구 제거

**현재**: 하단에 아래 경고 표시:
```
※ 이 화면을 닫으면 모든 응시 데이터가 삭제됩니다.
```

**변경 후**: `results.jsonl` 영속화 완료로 이 문구 제거. 대신:
```
결과가 인사팀에 자동 전송 및 저장되었습니다.
```

---

### 2-2. 시험 응시 화면 (`ExamScreen`) — 난이도 뱃지 숨김 확인

**현재**: 문항 카드 상단에 `diff` 뱃지(상/중/하) 표시 중.

**설계 §9.4**: "사원 화면엔 난이도 숨김" 명시.

**변경 후**: 난이도 뱃지 제거. 카테고리 뱃지만 유지.

---

### 2-3. 시험 시작 화면 (`IdentityScreen`) — 변경 없음

현재 완성도 높음. 변경 없음.

## 3. 신규 UI 요소 — 공통 컴포넌트 추가

### 3-1. StatusBadge 컴포넌트

```
draft      →  회색  "초안"
reviewing  →  노랑  "검토중"
approved   →  초록  "승인"
rejected   →  빨강  "반려"
```

기존 `Badge` 컴포넌트 확장으로 구현 (신규 type 추가).

### 3-2. FlagBanner 컴포넌트

```
warning       →  노랑 배너  "⚠ 카테고리·팀 불일치 — 확인 필요"
security_hold →  빨강 배너  "🔒 보안 키워드 감지 — 보안팀 확인 후 승인"
```

문항 카드 상단 또는 검토 카드 상단에 삽입.

### 3-3. ReasonCodeSelect 컴포넌트

난이도 수정 시 사유 코드 선택 드롭다운.

```
절차순서     / 기억단순     / 복합판단
근거약함     / 정답오류     / 현장중요도
용어정의     / 상황판단기초 / 근본원인분석
우선순위결정 / 텍스트불명확 / 기타
```

## 4. 변경하지 않는 것

| 항목 | 유지 이유 |
|---|---|
| 로그인 화면 전체 | 변경 불필요 |
| 사이드바 레이아웃 | 구조 유지, 검토 큐 항목만 빠른실행에 추가 |
| 사용자 승인 뷰 전체 | users.json 기반 유지, 변경 없음 |
| 결과 분석 뷰 전체 | 현재 mock → 2차에 Drive 연동 |
| 시험 시작/제출 확인 화면 | 완성도 충분 |
| 헤더·브레드크럼 구조 | 변경 없음 |

## 5. 변경 우선순위

```
P1 (1차 MVP 필수, 백엔드와 동시 작업)
  ├─ 문제 관리: 상태 뱃지 + 승인·반려 버튼
  ├─ 검토 뷰: 실제 reviewing 큐 연동
  ├─ 시험 생성: approved 문제만 출제 안내
  └─ 결과 화면: 경고 문구 → 저장 완료 문구

P2 (P1 완료 후)
  ├─ 대시보드 통계 카드 교체
  ├─ 시험 화면 난이도 뱃지 제거
  ├─ 사유 코드 드롭다운 (ReasonCodeSelect)
  └─ 설정 TODO 업데이트

P3 (2차 MVP)
  ├─ 결과 분석 Drive 연동
  └─ calibration 선별 UI
```
